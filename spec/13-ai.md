# 13 — AI, timing & sequencing

Navigation ships pathfinding and stops; this component covers *decisions* — when to chase, when to
give up, how a cutscene plays, how a cooldown works. The answer is the project's recurring one:
**the primitives already exist; AI is boring glue, not a new engine subsystem.** A behavior is a pure
transition over a blackboard ([`06`](06-things-behaviors.md)) — which is exactly a state machine, a
timer, and a sequencer, with no new construct.

> **Thesis:** a state machine is a blackboard `enum` + an exhaustive `match`; a timer is a `Fixed`
> countdown folded by `Time`; a scripted sequence is a `Timeline` value advanced each tick. All three
> are plain data and pure transitions — deterministic, serializable, unit-testable as folds, with no
> async, no coroutine, no decision DSL.

The hardest prior to resist is **async** ("wait N seconds, then act", `await`, a yielding coroutine):
hidden control flow and nondeterminism, the exact AX1 enemy. funpack replaces *waiting* with *state
advanced by the fixed tick* — you decrement a counter and `match` on it, never await.

---

## 1. Decisions — an `enum` + an exhaustive `match`

A `thing` carries its AI state as a blackboard `enum`; its behavior `match`es on that state and
returns the next. The match's **exhaustiveness is the totality guarantee** — every state handled or
the build fails, so an FSM can never fall through an unhandled state.

```funpack
enum Hunt { Patrol, Chase, Search }

behavior think on Hunter {
  fn step(self: Hunter, players: View[Player], time: Time) -> Hunter {
    let seen = visible(self.pos, players)        // Option[Vec2] — a pure perception predicate
    return match self.ai {
      Hunt::Patrol => patrol(self, seen)
      Hunt::Chase  => chase(self, seen)
      Hunt::Search => search(self, seen, time.dt)
    }
  }
}
```

Each state's logic is a **decomposed pure function** (tests in isolation, no world); the behavior is
just the dispatch. Reactive transitions ("flee on hit") are the same behavior also consuming a
`[Damaged]` signal and transitioning. Hierarchy is **sub-pipelines** ([`07`](07-pipelines.md)), not a
tree. There is **no `machine`/`state` keyword** — the `enum` is the state set, the `match` is the
transition function, exhaustiveness is the safety net.

## 2. Timing — a countdown is a field, not a scheduler

No `wait`, no `after(2s, …)`, no timer object with a callback — all async or hidden control flow. A
duration is a `Fixed` in the blackboard, decremented by `time.dt` and `match`ed at zero:

```funpack
fn seek(self: Hunter, dt: Fixed) -> Hunter {
  let t = self.search_t - dt
  if t <= 0.0 { return self with { ai: Hunt::Patrol, search_t: 0.0 } }
  return self with { pos: step_to(self.pos, self.last_seen, SPEED), search_t: t }
}
```

"Schedule X in N ticks" is a countdown initialized to `N`; "every M ticks" is a resetting counter; a
cooldown is the same field gating an action. Because the timestep is fixed and the countdown is
`Fixed`, **every machine fires on the same tick** — replay- and lockstep-stable by construction,
where a wall-clock timer would desync.

## 3. Sequencing — a `Timeline` is data, advanced each tick

The linear/scripted complement to the branching FSM (a cutscene, a spawn wave, an attack combo).
`engine.seq` ships a `Timeline`, the sequencing twin of `nav.path` — the engine owns ordering and the
elapsed-this-tick window; the body is a pure fold.

```funpack
data Step[A]     { at: Fixed, action: A }
data Timeline[A] { steps: [Step[A]], clock: Fixed = 0.0 }
fn advance(tl: Timeline[A], dt: Fixed) -> (Timeline[A], [A])   // actions whose time elapsed this window, in order
```

It is plain `data`, so a sequence is serializable — a save mid-cutscene resumes at the same step and
a replay re-fires identical actions on identical ticks. `Timeline[A]` is a **reserved stdlib generic**
parameterized by the game's own action type (user code authors no generics). A short 2–3 step beat is
just a cursor (`step: Int`, `step_t: Fixed`) `match`ed in a behavior; reach for `Timeline` when the
steps are many and data-shaped. **FSM for branching, Timeline for linear** — one obvious tool per
shape.

## 4. Selection & perception are pure folds

- **Selection / utility** — "the best target" is a `min_by_key`/`fold` scoring fold over a `View`
  (like arena's `nearest_player`). Utility AI is *scoring functions*, not an engine: there is no
  built-in subsystem that enumerates candidate actions and picks the highest-scoring one.
- **Perception** — "can I see the player" is a pure predicate (a range/cone test, or `nav.los`); it
  takes a `View` and returns a value, so taking it *is* the statement that this behavior senses.

Neither **utility AI** nor **GOAP** ships as a built-in — both are hidden decision machinery
competing with the authored `match`, the exact second decision structure §1 refuses. The shipped AI
surface is exactly three things: the blackboard `enum` matched exhaustively (the FSM), the `Fixed`
countdown ([§2](#2-timing--a-countdown-is-a-field-not-a-scheduler)), and `Timeline` sequencing
([§3](#3-sequencing--a-timeline-is-data-advanced-each-tick)). A **planner** — utility scoring over
weighted considerations, or a GOAP-style operator search toward a goal — is the **`extern` escape
hatch**: a library a behavior calls, which reads the blackboard, returns a chosen mode, and lets the
same exhaustive `match` dispatch on it. The planner lives *outside* the engine and writes back
through ordinary blackboard mutation; it is never a shipped engine subsystem.

## 5. Determinism & rejected priors

The whole AI is a pure fold over blackboard state — `think`/`patrol`/`search`/`run_waves` test by
exact equality with a constructed `View` and `Time.at(dt)`, no world. Synced blackboards reproduce
identical AI **with no AI-specific network messages**, and a failing tick becomes a permanent
regression via `capture_test` ([`28`](28-introspection.md)).

**Rejected** (each breaks an axiom): async/`await`/coroutines/`yield`; a behavior-tree/GOAP engine as
a built-in (a second decision structure — FSM + sub-pipelines cover the common case; a planner is an
escape hatch); wall-clock timers / `after(seconds, callback)`; `while`/unbounded loops in a decision.

AI state is a bare blackboard `enum` matched exhaustively — there is no declared transition graph,
no `machine`/`state` keyword, no behavior-tree/GOAP/utility-curve battery. `Timeline` carries linear
beats; branching is the FSM. A planner is the `extern` escape hatch.
