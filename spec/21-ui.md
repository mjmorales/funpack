# 21 — UI (`.fui`)

UI hallucination comes from scattered, hand-authored glue — markup, state, dispatch map, and routes
in four places an agent loses the thread across. funpack makes the glue **generated and
type-checked**.

> **Thesis: the binding name *is* the wire.** A `.fui` template — HTML-adjacent, single-file —
> declares what it *reads* and what it *emits*; the bake materializes **both ends as types** (a
> view-model `data` and a message `enum`); the compiler then forces the logic side to match.

A funpack UI *is* the Elm/React architecture, which is already the functional core: **Model** = a
view-model `data`; **View** = a pure `fn(model) -> View[Msg]`; **Messages** = signals; **Update** =
ordinary behaviors consuming those `Msg` signals. The template exists only to make authoring
single-file and wiring generated.

## 1. The `.fui` surface

```
screen Settings {
  col class="panel p-4 gap-3" {
    text class="text-2xl" { "Settings" }
    field placeholder="name" bind:value=player_name
    slider min=0 max=100 bind:value=volume
    row class="gap-2" {
      for p in volume_presets { button class="btn" @click=SetVolume(p.value) { "{p.value}" } }
    }
    button class="btn" @click=Back { "Back" }
  }
}
```

- **Widgets** are a closed ~14 set: layout `panel row col grid stack scroll spacer`; content `text
  image icon`; input `button field slider toggle select`. One obvious widget per job — no `div` soup,
  no user-defined template components.
- **Style** is space-separated tokens from the project **theme** — a closed, semantic-role vocabulary
  (`bg-panel`, `text-2xl`, `gap-4`, `rounded-md`; never raw values), checked like a `@gtag`; an
  unknown token is a compile error with a fix-it. The theme is `data`, so it is swappable behind
  unchanged templates.
- **Bindings** are the directives below — the whole point.

### Directives — each generates a typed edge

The data-flow triad names a direction of flow per sigil: `:attr=path` (**value in**, Vue bind),
`@event=Msg` (**message out**, Vue event), `bind:value=field` (**both**, Svelte two-way → lowers to a
read *and* a `Set`-message). Structure (`{ }`), control flow (`if`/`for`), and interpolation (`"{x}"`)
are funpack's own, reused — a `.fui` is "funpack you can see."

| In the template | Generates |
|---|---|
| `"{score}"`, `if game_over` | a view-model field (`score: Int`, `game_over: Bool`) |
| `:class=tone` | a typed read feeding the attribute (`tone: String`) |
| `@click=Pause` | a `Msg` variant (`Pause`) |
| `@click=SetVolume(p.value)` | a variant carrying the payload (`SetVolume(Int)`) |
| `bind:value=player_name` | **both** a read `player_name: String` *and* a `SetPlayerName(String)` variant |
| `for opt in difficulties { … }` | a list field whose **row type is inferred from the `opt.*` bindings used inside** |

The shapes are **inferred from usage — the template *is* the schema.** You never hand-declare the
view-model; you author the view and its data contract falls out. The surface is deliberately **not
JSX** (its power is arbitrary embedded logic — exactly what the seam forbids) and **not HTML tags**
(the widget set is closed and non-HTML, and `<>` is banned); it is funpack braces + a thin
declarative-directive layer, so only `@` / `:` / `bind:` are new tokens.

## 2. The generated seam & anti-hallucination

The bake emits committed funpack — the read contract, the write contract, and the screen function:

```funpack
data HudView { score: Int, time_left: Int, game_over: Bool }
enum HudMsg { Coin, Pause, Retry }
extern fn hud(model: HudView) -> View[HudMsg]
```

The game touches only this seam, in two pure pieces — a **projection** (`ui:` stage) building the
view-model and an **update** (interior stage) consuming the `Msg`. The payoff is mechanical: an agent
edits **only `hud.fui`**, adds `@click=Mute` → regeneration adds `HudMsg::Mute` → the update's `match`
is non-exhaustive → **compile error naming the exact behavior and missing variant**. Bind a new
`"{combo}"` → `HudView` gains `combo` → the projection no longer constructs → compile error naming
the missing field. The single-file edit **propagates as a typed error that says exactly what the
other domain must do** — the generated seam is the firewall keeping each task in one domain.

## 3. Routing — screens are values, navigation is a signal

No route config or string URLs. The set of `.fui` files generates an enum and a tagged message union
(`examples/hud/screens.gen.fun`):

```funpack
enum Screen { Hud, Pause, Settings }
enum AppMsg { Hud(HudMsg), Pause(PauseMsg), Settings(SettingsMsg) }
```

The active screen is sim state; the mount lifts each screen's view into the union with **`View.map`**
(the Elm `Html.map` primitive) — `hud(self.hud_view()).map(AppMsg::Hud)` — and the update unwraps the
tag and delegates. Adding a `.fui` extends both enums, so the mount *and* the update `match` stop
compiling until the screen is mounted and handled. **The screens *are* the route table**, so routing
cannot drift.

## 4. The `ui:` stage

`ui:` is a **distinct engine-closed stage** ([`06`](06-things-behaviors.md)), not a flavor of
`render:`: a `render:` behavior is output-only (`[Draw]`), but a `View[Msg]` **declares an inbound
edge** (the messages it can emit), which is categorically not a render behavior.

- A `ui:` behavior is a pure projection `fn(self) -> View[Msg]` with Render's input rules plus the one
  power to mount an interactive tree.
- **Compositing is stage order** — `ui:` runs after `render:`, so UI is over the scene by position;
  multiple `ui:` behaviors composite back-to-front in listed order.
- The **`Msg` round-trip is a deferred signal**: the engine hit-tests recorded pointer input against
  the composited tree and delivers each `Msg` next tick (the IO-result class). Effect closure requires
  every mounted `Msg` to be consumed somewhere; convention puts the handler in an early interior
  stage (`input: [on_msg] … ui: [view]`).

## 5. Determinism & rules

Layout and hit-testing are **deterministic fixed-point** (so "which button did this recorded click
hit" is replay-stable), but layout metrics are **never sim-readable** — UI's only sim-observable
output is the hit-tested `Msg` (there is no `width-of` API). UI state is `data` (serializable, part of
the save).

**Transitions & animation are engine visual-only polish off the determinism path.** Fades, slides, and
widget motion are **derived by the engine from view changes and theme tokens** — the engine diffs
successive `View`s and animates the delta under theme-supplied timing — and carry **no transition state
in the sim**: the projection emits the *target* view, never an in-flight tween. There is **no
imperative UI-animation authoring API** (no `animate(...)`, no timeline, no per-frame callback); like
mixing and device output in audio, animation is engine-side and not sim-observable, so a replay is
identical whether or not a transition was mid-flight.

**Accessibility is an engine rendering concern derived from the closed widget set + semantic theme
tokens.** UI scaling, contrast modes, and screen-reader labels fall out of the same two facts the
rest of UI rests on — every widget is one of the closed ~14 kinds (so the engine knows its role and can
expose a semantic label) and every style token is a semantic theme role (`bg-panel`, `text-2xl`, so the
engine can rescale or recontrast without touching templates). There is **no separate accessibility
authoring surface** — no aria attributes, no alt-text directive, no contrast flag in the `.fui`; an
author writes the closed widgets and semantic tokens, and accessibility is a rendering derivation
([`20`](20-render.md)), settings-driven like other per-machine preferences ([`24`](24-persistence.md)).

- **Expression sublanguage: paths and literals only** — no operators, calls, or ternaries; the one
  computation is string interpolation. Anything computed is a *named* view-model field set in the
  projection (so it is unit-testable): not `{score * 2}` but a field `doubled`; not `:class={sel ?
  …}` but `tone: String`.
- **Focus & gamepad** — focus is engine-managed deterministic state (a cursor over focusable widgets
  in document order, `focus-N` override). It is driven by the closed `UiAction` set
  (`stdlib/engine/ui.fun`: `NavUp`/`NavDown`/`NavLeft`/`NavRight`/`Confirm`/`Cancel`), bound in the
  *same* `bindings` function as gameplay input; **`Confirm` on a focused widget emits that widget's
  `@click` Msg**, so pointer, keyboard, and gamepad all reach one author-written binding.
- **List identity** — positional by default; `for item in items key=item.id` for stable identity on
  reorder/insert/delete (a `Ref[T]` is the natural key).
- **Payload typing** — an event payload is a path; primitives infer, a domain type is named once with
  an inline ascription (`for opt in difficulties: { id: Ref[Difficulty] } { … }`).
- **Enum variant as a function value** (a general language fact surfaced by `.map`): `AppMsg::Hud`
  denotes `fn(HudMsg) -> AppMsg`; a nullary variant is the value itself.

**Incremental replacement:** the seam (`HudView`, `HudMsg`, `fn hud`) is the `@stub`-for-UI analog —
a `.fui` template can be replaced by a hand-authored `engine.ui`-builder view (`panel`/`text`/
`button`/`class`/`when`/`map`) behind the same seam (`backing: template("hud.fui")` →
`funpack(hud_view)`), with no change to the projection or update.
